# Blog 寫作指南

這份指南用於撰寫部落格文章，確保內容對讀者有實際幫助，同時兼顧 SEO 與 AEO（Answer Engine Optimization）。

---

## 專案基本資訊

### 分類

只有兩種分類，必須二擇一：

- `ai` - AI 相關：工具、新聞、教學、趨勢分析
- `game` - 遊戲相關：遊戲推薦、產業新聞、開發知識

### 檔案結構

```
src/content/blog/
└── YYYY/
    └── MM/
        └── DD/
            └── article-slug.md
```

**範例：** `src/content/blog/2026/01/22/claude-code-ai-development.md`

### 檔名規範

- 使用英文小寫
- 單字之間用 `-` 連接
- 簡短但能描述文章主題
- 避免特殊字元和空格

## Frontmatter 設定

### 完整模板

```yaml
---
title: "文章標題"
description: "50-160 字的摘要，會用於列表頁和 SEO"
category: "ai" # 或 "game"
pubDate: 2026-01-22
updatedDate: 2026-01-23 # 選填，更新時才加
image: "https://images.unsplash.com/photo-xxx?w=1200"
imageAlt: "圖片描述文字"
tags: ["標籤1", "標籤2", "標籤3"]
draft: false # true 則不會發布
---

```

### 欄位說明

| 欄位        | 必填 | 說明                       |
| ----------- | ---- | -------------------------- |
| title       | 是   | 文章標題，清楚具體         |
| description | 是   | 摘要，50-160 字為佳        |
| category    | 是   | 只能是 `ai` 或 `game`      |
| pubDate     | 是   | 發布日期 `YYYY-MM-DD`      |
| updatedDate | 否   | 更新日期，有修改時才加     |
| image       | 否   | 封面圖 URL，建議 1200px 寬 |
| imageAlt    | 否   | 圖片替代文字（無障礙/SEO） |
| tags        | 否   | 標籤陣列，3-5 個為佳       |
| draft       | 否   | 預設 false，草稿設 true    |

> **[AI 助理重要提醒]**
> 雖然 `image`, `imageAlt`, `draft` 等欄位被標記為非必填，但為了確保網站內容的完整性和一致性，**請在建立每一篇文章時，都務必包含這幾個欄位**。
>
> - `image`: 必須提供一個圖片 URL，如果沒有合適的，可先使用佔位圖。
> - `imageAlt`: 必須提供有意義的圖片描述。
> - `draft`: 應明確設為 `false`，以確保文章能被正常發布。

## 文章結構

> **[AI 助理重要提醒]**
> 撰寫文章時，**必須嚴格遵守下方的「必要元素」清單**。每一篇文章都應包含從「標題」到「重點整理」的所有區塊，不可遺漏。這能確保所有產出內容的結構完整、品質一致。

### 必要元素

#### 1. 標題

- 清楚、具體、帶有結果或價值
- 好：`Claude Code 實戰：AI 輔助開發真的能取代工程師嗎？`
- 差：`關於 AI 的一些想法`

#### 2. 開頭

直接切入主題，不需要「這篇給誰看」的段落。讓 description 欄位負責說明目標讀者即可。

#### 3. 核心內容

- 使用 H2/H3 分段，結構清晰
- 解釋概念 → 實際步驟 → 具體範例 → 取捨/限制
- 每段都要帶來新資訊

#### 4. FAQ（3-5 題）

- 回答讀者最可能有的疑問
- 答案簡短直接
- 對 AEO（語音搜尋、AI 搜尋）有幫助

#### 5. 參考資料（2-5 個）

- 附上連結（系統將自動設定為開新視窗，無需手動處理）
- 簡短說明為什麼這個資料有用

#### 6. 重點整理（3-6 點）

- 可快速掃讀的摘要
- 動作導向，告訴讀者下一步
- 不需要結尾的留言邀請（本站無留言功能）

### 加分元素（選用）

- 「常見錯誤」段落
- 「什麼情況不適用」說明
- 快速檢查清單
- 比較表格
- 程式碼範例（如適用）

## 寫作原則

### 核心心法

1. **從讀者出發** - 他是誰？遇到什麼問題？想做什麼決定？
2. **提供具體價值** - 步驟、範例、數據、取捨分析
3. **不廢話** - 每段都要有新資訊，砍掉裝飾性文字
4. **附上來源** - 有數據或事實宣稱就要引用
5. **誠實面對限制** - 不只講優點，也講缺點和不適用情況

### 語氣風格

- **專業但不學術** - 像資深同事在解釋，不是教科書
- **直接但不粗魯** - 講重點，不繞圈子
- **客觀但有立場** - 可以給建議，但要說明理由
- **中文為主** - 專有名詞可保留英文，但避免中英夾雜過多

### 避免

- 空泛的形容詞（「超級厲害」「非常好用」）
- 沒有根據的宣稱
- 過度行銷語氣
- AI 味太重的制式句型
- 不必要的 emoji

### 一致性檢查 (Consistency)

在撰寫新文章前，**請務必閱讀 1-2 篇現有的部落格文章**（位於 `src/content/blog/`）。

- **比對格式**：確認標題層級、列表樣式、引用格式是否與現有文章一致。
- **模仿語氣**：感受現有文章的「說話方式」，保持部落格風格統一。
- **參考結構**：觀察其他文章如何安排開頭、結尾和 FAQ。

## AI 輔助寫作提示詞

當使用 AI 輔助撰寫文章時，可使用以下提示詞模板：

```
你是一位資深的科技/遊戲記者，正在為實際讀者撰寫一篇實用的部落格文章。

目標：幫助讀者解決具體問題並做出決定
語氣：清晰、直接、像人類專家、不浮誇、不行銷

主題：{topic}
目標讀者：{persona}
讀者痛點：{pain}
期望成果：{outcome}
字數限制：{length}

必要結構：
1) 標題（具體、承諾結果）
2) 開頭：2-4 句（問題 + 對象 + 成果）
3) 核心內容（H2/H3）：
   - 解釋概念
   - 步驟流程
   - 實際範例
   - 取捨與限制
4) FAQ（3-5 題，答案簡短）
5) 參考資料（2-5 個連結，說明為何有用）
6) 重點整理（3-6 點，不重複）

規則：
- 不填充、不堆砌流行語
- 每段必須帶來新資訊
- 盡量用具體數字、工具、步驟
- 有宣稱就附來源
- 短句、避免誇大
- 輸出 Markdown 格式
- 分類只能是 ai 或 game
```

### 提示詞範例（填入版）

```
主題：2026 年最值得關注的 5 款獨立遊戲
目標讀者：想找新遊戲玩的玩家
讀者痛點：Steam 遊戲太多，不知道選什麼
期望成果：找到符合自己口味的遊戲
字數限制：1500-2000 字，需包含每款遊戲的特色比較
```

## 寫作檢查清單

發布前確認：

### 內容品質

- [ ] 開頭有明確說「給誰看、能解決什麼」
- [ ] 有清楚的步驟或流程可以跟著做
- [ ] 有參考現有文章的格式與語氣，保持風格一致
- [ ] 有至少一個具體範例
- [ ] 有提到缺點、限制或不適用情況
- [ ] 有數據或事實的地方都有附來源
- [ ] 結尾有可快速掃讀的重點整理

### 格式規範

- [ ] Frontmatter 必填欄位都有填
- [ ] category 是 `ai` 或 `game`
- [ ] 檔名使用英文小寫和 `-`
- [ ] 圖片有填 imageAlt
- [ ] 標題沒有用問號以外的標點結尾

### SEO/AEO

- [ ] 標題包含主要關鍵字
- [ ] description 在 50-160 字之間
- [ ] 有 FAQ 段落（幫助語音搜尋/AI 搜尋）
- [ ] 有使用 H2/H3 結構化內容

## 文章類型參考

### 新聞類

- 報導事件、發布、更新
- 重點：時效性、來源可信度、影響分析
- 範例：`NVIDIA 發表新顯卡：RTX 5090 規格、價格與上市資訊`

### 教學類

- 教讀者如何做某件事
- 重點：步驟清晰、有截圖/程式碼、常見問題
- 範例：`用 Stable Diffusion 生成遊戲角色：從安裝到出圖完整教學`

### 分析類

- 深入探討某個主題或趨勢
- 重點：數據支撐、多角度觀點、有自己的見解
- 範例：`AI NPC 會改變遊戲產業嗎？技術現況與未來展望`

### 推薦類

- 推薦工具、遊戲、資源
- 重點：實測心得、優缺點比較、適合對象
- 範例：`2026 年 5 款最佳 AI 繪圖工具比較：Midjourney vs DALL-E vs Stable Diffusion`
